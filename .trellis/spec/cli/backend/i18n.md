# Internationalization Contract

> Long-term contract for maintaining Trellis as an English and Chinese product without splitting the codebase.

---

## Scenario: CLI and Template Localization

### 1. Scope / Trigger

- Trigger: any new or changed CLI/user-facing text, locale-aware generated
  template, docs-site page, or release gate.
- Goal: Trellis must support English and Simplified Chinese without translating
  protocol identifiers or making generated templates depend on host machine
  locale.

### 2. Signatures

- `trellis --locale en|zh --help`
- `trellis --lang en|zh --help`
- `trellis init --locale en|zh [options]`
- `trellis update --locale en|zh [options]`
- `trellis uninstall --locale en|zh [options]`
- `TRELLIS_LANG=en|zh` or `TRELLIS_LOCALE=en|zh`
- `.trellis/config.yaml`: `language: en|zh`
- `pnpm --filter @mindfoldhq/trellis check:i18n`
- `node packages/cli/scripts/check-i18n.js --release`

### 3. Contracts

- CLI locale option accepts only `en`, `zh`, or regional aliases that normalize
  to those values, such as `en-US` or `zh-CN`.
- CLI locale selection precedence is:
  1. command option
  2. environment
  3. project config
  4. `en`
- Runtime messages are looked up by semantic key with `{name}` placeholders.
- `en.ts` is the canonical runtime message key set; `zh.ts` must satisfy the
  same keys.
- Locale-aware markdown templates use same-name siblings:
  `agents.md` -> `agents.zh.md`.
- Missing localized template siblings fall back to English.

### 4. Validation & Error Matrix

| Condition | Expected behavior |
| --- | --- |
| Unsupported CLI locale value | Throw `Unsupported locale "<value>". Use "en" or "zh".` |
| Missing Chinese runtime message key | TypeScript compile or `check:i18n` fails |
| Extra Chinese runtime message key | `check:i18n` fails |
| docs-site missing in normal check | Warn and skip docs-site checks |
| docs-site missing in release check | Fail release check |
| English docs-site page missing `zh/` mirror | Fail release check |
| English-only protocol identifier | Keep unchanged in every locale |

### 5. Good/Base/Bad Cases

- Good: `trellis init --locale zh` writes `language: zh`, Chinese
  `AGENTS.md`, and Chinese bootstrap/joiner PRDs while preserving commands and
  YAML keys in English.
- Base: `trellis init` with no locale option, env, or config writes English
  output regardless of system `LANG`.
- Bad: translating `task.json.status` values or `.trellis/workflow.yaml` keys.
- Bad: using an English display sentence as a message key.

### 6. Tests Required

- Unit: `normalizeLocale`, `resolveLocale`, `readProjectLocale`, and placeholder
  formatting.
- Integration: `init({ locale: "zh" })` writes localized config and generated
  guidance.
- CLI smoke: built `dist/cli/index.js --locale zh --help` for root, `init`,
  `update`, and `uninstall`.
- Gate: `check:i18n` passes in normal mode; release scripts run
  `check-i18n.js --release`.

### 7. Wrong vs Correct

#### Wrong

```ts
messages["Update complete!"] = "更新完成！";
```

#### Correct

```ts
t("update.complete", { action, projectVersion, cliVersion });
```

#### Wrong

```yaml
状态: done
```

#### Correct

```yaml
status: done
```

## Supported Locales

Trellis supports two product locales:

| Locale | Meaning | Role |
| --- | --- | --- |
| `en` | English | Default locale and protocol language |
| `zh` | Simplified Chinese | First-class user-facing locale |

Locale selection order is:

1. CLI option: `--locale en|zh`
2. Environment: `TRELLIS_LANG` or `TRELLIS_LOCALE`
3. Project config: `.trellis/config.yaml` `language: en|zh`
4. Fallback: `en`

Trellis intentionally does **not** auto-switch based on system `LANG`. Generated
templates are hash-tracked by `trellis update`; making them depend on host
locale would make the same project initialize differently on different machines.

---

## Must Be Translated

The following surfaces are user-facing and must have English and Chinese coverage:

- Commander help text for `trellis`, `trellis init`, `trellis update`, and `trellis uninstall`
- Human-readable CLI status, warning, error prefix, and summary strings
- Root project guidance generated by `trellis init`, especially `AGENTS.md`
- Bootstrap and joiner onboarding PRDs generated by `trellis init`
- docs-site MDX pages and `docs.json` navigation entries, when the docs-site submodule is initialized
- Changelog pages used by release scripts

---

## Must Stay Stable / English

These are protocols, identifiers, or compatibility surfaces. Do not translate them:

- Command names and flags: `init`, `update`, `uninstall`, `--locale`, `--migrate`
- JSON / YAML keys: `language`, `packages`, `default_package`, `status`, `dev_type`
- Workflow state names and task status values: `planning`, `in_progress`, `done`
- File and directory names: `.trellis/`, `workflow.yaml`, `task.json`, `implement.jsonl`, `check.jsonl`
- Agent IDs, skill IDs, platform IDs, package names, npm package names
- Code blocks, shell commands, import paths, and API field names

Rule: translate prose around the protocol, not the protocol itself.

---

## Message Catalog Rules

Runtime CLI strings live under `packages/cli/src/i18n/messages/`.

- English (`en.ts`) defines the canonical key set.
- Chinese (`zh.ts`) must satisfy the exact English key set via TypeScript.
- Keys are semantic identifiers, not display text.
- Never use the English sentence as a translation key.
- Variable placeholders use `{name}` and must be preserved in every locale.

Example:

```ts
t("update.complete", { action, projectVersion, cliVersion });
```

---

## Template Rules

Locale-aware templates use same-name sibling files:

```text
agents.md
agents.zh.md
```

When a localized sibling exists, `trellis init --locale zh` and `trellis update`
use it. When it does not exist, Trellis falls back to English.

This fallback is intentional:

- It allows incremental translation without blocking releases.
- It keeps protocol-heavy agent templates stable until translated and reviewed.
- It prevents half-translated protocol files from breaking generated projects.

When adding a high-value user-facing template, add the `.zh` sibling in the
same change. When adding a protocol-heavy template, English fallback is allowed
until a reviewed Chinese version exists.

---

## Docs-Site Rules

The docs-site submodule mirrors English pages under `zh/`.

- New user-facing MDX pages must be added in both English and Chinese.
- `docs.json` navigation must include both language trees.
- Code blocks and identifiers stay unchanged across locales.
- Release changelogs require both `changelog/v<version>.mdx` and
  `zh/changelog/v<version>.mdx`.

Run:

```bash
pnpm --filter @mindfoldhq/trellis check:i18n
```

Use release mode in release scripts:

```bash
node scripts/check-i18n.js --release
```

---

## Tests and Gates

Required checks:

- TypeScript compile verifies message key parity.
- Unit tests cover locale normalization and message formatting.
- Init integration tests cover Chinese project config and generated guidance.
- `check-i18n.js` verifies docs-site bilingual structure when available.
- Release scripts run `check-i18n.js --release` before tagging.

Do not merge a new user-facing English-only CLI message without adding the
matching Chinese message.
